# Deployment Guide

Target topology: **Frontend → Vercel**, **API → Render**, **Database → Neon (Postgres)**.

The architecture is host-agnostic — the API is a standard Dockerised Node service and the
frontend is a static Vite build — so Railway/Fly.io/Netlify work with the same steps.

---

## 1. Database — Neon

1. Create a project at [neon.tech](https://neon.tech) → you get a connection string.
2. Use the **pooled** connection string (host contains `-pooler`) and keep `?sslmode=require`:
   ```
   postgresql://USER:PASSWORD@ep-xxx-pooler.REGION.aws.neon.tech/DB?sslmode=require
   ```
   The pooler is important on serverless/auto-scaling platforms to avoid exhausting connections.
3. This single value becomes `DATABASE_URL` for the API.

---

## 2. Backend — Render

You can deploy via the included **Blueprint** (`backend/render.yaml`) or manually.

### Option A — Blueprint (recommended)
1. Push this repo to GitHub.
2. Render → **New + → Blueprint** → select the repo. It reads `backend/render.yaml` and
   provisions a Docker web service with a health check on `/health`.
3. Fill the `sync:false` env vars (see below). `JWT_*` secrets are auto-generated.

### Option B — Manual (Docker)
- **Root directory:** `backend`
- **Environment:** Docker (uses `backend/Dockerfile`)
- **Health check path:** `/health`
- The Dockerfile runs `prisma migrate deploy` on boot, then starts the server.

### Required environment variables (API)

| Variable | Value |
| --- | --- |
| `NODE_ENV` | `production` |
| `PORT` | `4000` (Render injects `PORT`; the app reads it) |
| `DATABASE_URL` | Neon pooled URL from step 1 |
| `JWT_ACCESS_SECRET` / `JWT_REFRESH_SECRET` | long random strings (auto-generated by the blueprint) |
| `CLIENT_URL` | your Vercel URL, e.g. `https://vutto-auctions.vercel.app` (**no trailing slash**) |
| `API_PUBLIC_URL` | the Render URL, e.g. `https://vutto-auctions-api.onrender.com` |
| `COOKIE_SECURE` | `true` |
| `SEED_ADMIN_EMAIL` / `SEED_ADMIN_PASSWORD` | your admin credentials |

> `CLIENT_URL` must exactly match the deployed frontend origin — it drives both CORS and the
> Socket.IO CORS policy.

### First deploy: migrate & seed
- **Migrations** run automatically (`prisma migrate deploy` in the Dockerfile `CMD`).
- **Seed** (optional, once) from the Render **Shell**: `npm run db:seed`.

---

## 3. Frontend — Vercel

1. Vercel → **New Project** → import the repo.
2. **Root directory:** `frontend`. Framework preset: **Vite** (build `npm run build`, output `dist`).
   `frontend/vercel.json` adds the SPA rewrite so client-side routes don't 404 on refresh.
3. **Environment variables:**

   | Variable | Value |
   | --- | --- |
   | `VITE_API_URL` | `https://vutto-auctions-api.onrender.com/api` |
   | `VITE_SOCKET_URL` | `https://vutto-auctions-api.onrender.com` |
   | `VITE_APP_NAME` | `Vutto Auctions` |

4. Deploy. Then set the API's `CLIENT_URL` to this Vercel URL and redeploy the API.

---

## 4. Cross-site cookie checklist (important)

Because the frontend and API are on **different domains**, the refresh cookie must be
cross-site capable. This is already handled in code when `COOKIE_SECURE=true` /
`NODE_ENV=production`:

- cookie flags: `httpOnly; Secure; SameSite=None`
- CORS: `origin: CLIENT_URL`, `credentials: true`
- Axios: `withCredentials: true`

If login works but you're logged out on refresh, it's almost always a `CLIENT_URL` mismatch
or `COOKIE_SECURE` not set to `true` in production.

## 5. Production build sanity (local)

```bash
# Backend
cd backend && npm run build && node dist/server.js

# Frontend
cd frontend && npm run build && npm run preview
```

---

## 6. CI suggestion

A minimal pipeline:

```yaml
# backend
- npm ci && npx prisma generate && npm run build && npm test
# frontend
- npm ci && npm run build
```

Run the integration suite against an ephemeral Postgres with `RUN_DB_TESTS=1`.
